## Conventions

### File and component naming
- React components: `PascalCase.tsx` (e.g., `TransactionEntryForm.tsx`, `ActivityRow.tsx`)
- Hooks: `camelCase` prefixed with `use` (e.g., `useAuthState.ts`, `useTransactionPipeline.ts`)
- Utilities / lib: `camelCase` (e.g., `formatCurrency.ts`, `dateUtils.ts`)
- Constants: files named descriptively (`constants.ts`, `errorMessages.ts`); values in `UPPER_SNAKE_CASE`
- Test files: `*.test.tsx` / `*.test.ts` (none currently exist in frontend; pattern is defined in CODING_STANDARDS)
- Backend classes: `PascalCase`; packages lowercase by domain segment (`com.kaizen.backend.<feature>`)
- Backend DTOs: `<Action><Entity>Request` / `<Entity>Response` (e.g., `BudgetCreateRequest`, `CategoryResponse`)
- Backend exceptions: `<Entity>NotFoundException`, `Duplicate<Entity>Exception`

### Module / import style
- ESM throughout (`"type": "module"` in `package.json`)
- No path aliases configured in `tsconfig.app.json`; all imports are relative
- `verbatimModuleSyntax: true` — type-only imports must use `import type`
- Barrel exports used at feature level (e.g., `features/insights/index.ts`)

### TypeScript rules
- `strict: true`; no `any` in app code; prefer `unknown` + type guards
- Avoid `enum`; prefer string unions
- Explicit return types required on exported functions and public APIs
- `noUnusedLocals` and `noUnusedParameters` are compiler errors, not warnings

### React patterns
- Feature-first organization under `features/*`; `shared/` only for truly reusable things
- Keep components pure; derive UI from props/state
- Side effects only in `useEffect` / RTK Query hooks
- `useMemo`/`useCallback` only when measured and justified
- All data-fetching components must handle loading, error, and empty states
- Lazy loading (`React.lazy`) is used for all secondary/feature pages in the router

### Commit message style
- Conventional Commits format — confirmed from recent git log:
  `fix(transactions): ...`, `chore(transactions): ...`
- Scope is the feature name in parentheses

### Branching model
- Single `main` branch; PRs target `main` (confirmed from workflow `on: pull_request: branches: [main]`)

### PR template
- Located at `.github/pull_request_template.md`
- Sections: Description (1–2 sentences), Key technical implementations (bullet list), checklist: Bug fix / New Feature / Documentation update / Other

### API documentation policy (backend)
- Every endpoint addition or change must update OpenAPI/Swagger docs, Postman collection, and examples in the same PR — endpoint work is not complete without this

### Environment variables
- Frontend: `VITE_API_BASE_URL` (required; validated at runtime in `baseApi.ts`)
- Template at `frontend/.env.example`; actual `.env` is gitignored
- Backend: all env vars defined via Spring `${VAR:default}` syntax; template at `backend/.env.example` (Unknown — needs human confirmation of exact file name)

### Behavior
- Renders a `+` toggle button (bottom-right, fixed position: `bottom-24 right-6` on mobile, `bottom-10 right-10` on desktop)
- Toggling open animates four action buttons upward (opacity + translateY + scale transition, 300ms)
- On open: toggle button rotates 45° (becomes a close/X icon)
- On mobile (`max-width: 768px`, via `useMediaQuery`): action labels animate in with staggered 50ms delay per item
- On desktop: action labels appear on `group-hover` only
- Calls `navigator.vibrate?.(8)` on toggle (haptic feedback, mobile only, gracefully ignored on desktop)
- All action buttons close the dial after invoking their callback
- `z-index: z-40`; `pointer-events-none` on wrapper, `pointer-events-auto` on individual buttons

### TypeScript Types

#### `BudgetResponse` (from `budgetApi.ts`) — the live/server budget shape
```ts
interface BudgetResponse {
  id: number
  userId: number
  categoryId: number
  categoryName: string
  amount: number          // allocated budget ceiling
  expense: number         // actual spending so far this period
  burnRate: number | null         // $/day, null if daysElapsed < 3
  dailyAllowance: number | null   // remaining budget / days left, null if insufficient data
  projectedTotal: number | null   // estimated end-of-period spend, null if insufficient data
  daysElapsed: number | null
  daysLeft: number | null
  period: BudgetPeriod            // 'MONTHLY' | 'WEEKLY'
  createdAt: string               // ISO date string
  updatedAt: string               // ISO date string
}
```

Note: `BudgetResponse` does NOT have an `archived` field. `BudgetsPage` casts to `BudgetResponse & { archived?: boolean }` to future-proof the Active/Archived tab UI, treating all current budgets as active.

#### Data / state management

**RTK Query hooks consumed:**
- `useGetBudgetsQuery()` — pre-loads existing budgets; used to hydrate `pendingBudgets` slice on first visit and to compute `existingExpenseByCategory` (a `Map<categoryId, expense>`) for outstanding balance calculations
- `useSaveSmartBudgetsMutation()` — management-flow "Done" path: `POST /budgets/batch`
- `useCompleteOnboardingMutation()` — onboarding "Done" path (from `authApi`)
- `useSkipBudgetSetupMutation()` (inside `SkipBudgetTrigger`) — `PUT` to mark skip preference

**Redux selectors read:**
- `selectPendingBudgets` → `sessionBudgets: PendingBudget[]` — the in-progress draft list
- `selectBalanceValue` → `reduxBalance: number | null` — falls back to `user.balance`
- `selectFundingSourceType` → required before onboarding completion

**Redux actions dispatched:**
- `setPendingBudgets` — one-time hydration from `existingBudgets` on first visit (guarded by `hasHydratedRef`)
- `addPendingBudget` — appends new draft
- `updatePendingBudget` — replaces existing draft by `categoryId`
- `removePendingBudget` — removes draft by `categoryId` (no undo by design; comment references `UNDO_POLICY.md`)

**Direct REST call (not RTK Query):**
- `getCategories()` from `features/categories/api` — called in a `useEffect` on mount; result stored in local `categories` state. This is a direct fetch, not RTK Query. Loading/error state managed locally (`isLoadingCategories`, `categoryError`).
- `createCategory()` from `features/categories/api` — called during custom category creation inside `handleAddBudgetClick`; result appended to local `categories` state.

**Key derived values (all via `useMemo`):**
- `disabledCategoryIds` — `sessionBudgets.map(b => b.categoryId)` — prevents double-adding same category
- `existingExpenseByCategory` — `Map<categoryId, expense>` built from `existingBudgets`
- `totalAllocated` — sum of `outstandingFor(budget)` = `max(budget.amount - existingExpense, 0)` across all session budgets
- `allocatedRaw` — sum of `budget.amount` (gross, ignoring existing spend)
- `unallocated` — `balance - totalAllocated`
- `allocationPercentage` — `Math.round((allocatedRaw / balance) * 100)`
- `isOverAllocated` — `unallocated < 0`
- `invalidBudgetIds` — `Set` of `categoryId`s where `budget.amount <= 0` (triggers red highlight on `BudgetCard`)
- `nextCustomCategoryDesign` — `getAutoAssignedCategoryDesign(userCategories)` — auto-picks icon+color for next custom category
- `availableForCurrent` — balance minus other outstanding budgets plus current budget's existing spend (used as Input `max`)

**Validation logic (all in `handleAddBudgetClick`):**
1. Category must be selected (not `''`)
2. Amount must be a finite number ≥ 1
3. Amount must not exceed `availableForCurrent` (computed per-edit accounting for existing spend)
4. If `CUSTOM_CATEGORY_OPTION_VALUE`: custom name must be non-empty and not duplicate an existing category name (case-insensitive check)
5. Custom category is created via `createCategory()` API call before the budget is dispatched to Redux

**Hydration guard:** `hasHydratedRef` (a `useRef<boolean>`) prevents re-hydrating `pendingBudgets` from the server after the user has started editing in the session.

#### Edit and Delete actions — how they work

**Edit:**
- `BudgetCard` renders a plain `<button type="button">Edit</button>` (not the shared `Button` component)
- Clicking Edit calls `onEdit` prop → `openModal(budget)` in `ManualBudgetSetupPage`
- `openModal` pre-fills modal state: `selectedBudgetId = budget.categoryId`, `selectedCategoryId = budget.categoryId`, `amountInput = budget.amount.toString()`, `selectedPeriod = budget.period`
- Modal title becomes "Edit budget"; confirm button becomes "Update"
- On submit, `updatePendingBudget` is dispatched (replaces the entry at matching `categoryId` in Redux)
- The edited category is excluded from `disabledIds` so it remains selectable in the picker

**Delete:**
- `BudgetCard` renders a plain `<button type="button">Delete</button>` (not the shared `Button` component)
- Clicking Delete calls `onRemove` prop → `handleDeleteBudget(budget.categoryId)` in `ManualBudgetSetupPage`
- Dispatches `removePendingBudget(categoryId)` — immediate, no confirmation dialog, no undo
- No `DestructiveActionDialog` is used here (unlike `SessionsPage` and the transactions feature)
- Comment in source: "No undo by design — see UNDO_POLICY.md. (Session-local removal; nothing is persisted yet.)"

#### Button styling on BudgetCard (observed inconsistency)

Both Edit and Delete buttons in `BudgetCard` are raw `<button>` elements styled with Tailwind utility classes, NOT the shared `Button` component:

```
Edit button:
  rounded-xl px-4 text-sm font-semibold text-text-secondary
  hover:bg-surface-secondary hover:text-text-primary active:scale-95
  + fluidLayout.touchTarget (min-height 44px on mobile)
  + sm:h-9 sm:min-h-0 sm:px-3 sm:py-2

Delete button:
  rounded-xl px-4 text-sm font-semibold text-text-secondary

  + sm:h-9 sm:min-h-0 sm:px-3 sm:py-2
```

The rest of the app uses `<Button variant="ghost">` for edit actions and `<Button variant="destructive">` (or `variant="ghost"` with destructive intent) for delete actions. The raw `<button>` usage here is inconsistent with the shared design system and means these buttons do not pick up any future `Button` component changes (loading states, focus ring styles, etc.).

## Constraints

### Mandatory (from AGENTS.md and CODING_STANDARDS.md)

**General**
- CODING_STANDARDS.md is the single authoritative standard. It overrides all default style preferences and AI defaults.
- Clarity over cleverness; small, focused functions; no silent failures; no hidden global state.
- Security by default: validate all inputs; least privilege; never log secrets, tokens, or full PII payloads.

**Frontend — typography (CODING_STANDARDS.md §1.7.1)**
- Use only Tailwind CSS utilities for typography. No inline styles for typography.
- Use only the 11 approved roles: `display`, `h1`, `h2`, `h3`, `h4`, `body-lg`, `body`, `body-sm`, `label`, `caption`, `code`. Do not invent new roles.
- Use semantic HTML elements before styling (e.g., `<h1>` not `<div>` for page titles).
- Text color: exactly three neutral levels only — `text-foreground` (primary), `text-muted-foreground` (secondary), `text-subtle-foreground` (metadata). No `text-black`, `text-white`, `text-gray-500`, or any arbitrary color utilities.
- On filled colored surfaces use the matching `text-on-*` token (e.g., `text-on-primary`). NOTE: `text-on-primary` does not currently exist in the CSS — this is an open gap.
- `tracking-tight` on headings only. Only `font-normal`, `font-medium`, `font-semibold` allowed.
- Do not use arbitrary font sizes (e.g., `text-[15px]`) unless explicitly required.
- Only headings should usually scale responsively.

**Frontend — color and accessibility (CODING_STANDARDS.md §1.7.1 + AGENTS.md)**
- Target WCAG AAA for text content where practical (minimum 7:1 for normal text, 4.5:1 for large text).
- Borders, focus indicators, icons, and non-text controls must meet at least 3:1.
- Treat any filled colored surface as incomplete until base, hover, and active states are contrast-checked.
- Do not assume `text-foreground` is safe on a colored background without verification.
- When changing theme tokens, verify both light and dark mode in the same change.
- Do not use `hover:bg-black/5` or similar hardcoded opacity overlays — these break dark mode. (Known systemic issue flagged in prior audits.)

**Frontend — forms (CODING_STANDARDS.md §1.7.2)**
- Every field must have a visible label; placeholders cannot replace labels.
- Validation errors appear on `blur` or `submit` only — never during typing.
- Required attributes: `id`, `htmlFor`, `aria-invalid`, `aria-describedby`.
- Minimum input height: `h-11` (44px) for mobile touch targets.
- Do not use `type="number"` for phone, IDs, OTP, or postal codes.

**Frontend — design tokens**
- All color, spacing, and radius values must reference the design token system, not hardcoded palette values.
- The Tailwind `@theme inline {}` block in `frontend/src/shared/styles/index.css` is the single source of truth for what tokens are available to Tailwind utility classes.
- CSS variables defined in `globals.css` but NOT re-exported in `@theme inline` are NOT available as Tailwind utilities (they can only be referenced via `var()` in custom CSS or inline styles).

**Frontend — icon system**
- All icons must be rendered via `<SharedIcon>` — never import Lucide icons directly in feature or shared components.
- Exception observed and scoped: `YourAccountPage.tsx`, `AppearancePage.tsx`, and `SessionsPage.tsx` use inline SVGs instead of `SharedIcon`. This is an existing pattern in the account feature.
- To add a new icon: add the Lucide import to `IconConstants.ts` and register it in `UI_ICONS` (or `PAYMENT_METHOD_ICONS`); then reference it by string name via `SharedIcon`.

**Frontend — dark mode**
- Dark mode is toggled by adding class `dark` and `data-theme="dark"` to `<html>` (via ThemeProvider).
- The `@custom-variant dark` in `index.css` is `(&:where(.dark, .dark *))` — dark variant applies when `.dark` is an ancestor.
- System preference (`prefers-color-scheme`) is respected when theme is set to `"system"`.
- All UI changes must be tested in both light and dark themes.

**Backend — architecture**
- Controllers: HTTP boundary only (no business logic, no direct repo calls).
- Services: business logic and `@Transactional`.
- DTOs: never expose JPA entities directly in responses.
- All error responses use one of two envelopes: `ErrorResponse` (general) or `ValidationErrorResponse` (field validation).
- Avoid `EAGER` fetch; use `LAZY` + explicit fetch joins/entity graphs.
- Prefer pagination for all list endpoints.

**Backend — database migrations (CODING_STANDARDS.md §2.10)**
- Committed migration scripts are read-only — never edit existing scripts.
- All schema changes must be new, higher-numbered scripts (e.g., `V5__Add_Index.sql`).
- Use `IF NOT EXISTS` for columns/indexes during development-heavy phases.
- No `repair-on-migrate` in production.

**Backend — security**
- Never embed secrets in config files.
- Validate and sanitize all path params, query params, and request bodies.
- Use CSRF protection where applicable.
- Require authentication/authorization for all protected endpoints.

**API documentation**
- Any backend endpoint addition or change is not complete until OpenAPI/Swagger, Postman collection, and examples are updated in the same PR.

**Environment variables**
- When adding or changing env var usage in backend, frontend, Docker, or workflows, update the corresponding `.env.example` in the same change.
