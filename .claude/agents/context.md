## Conventions

### Naming Patterns (Frontend)
- React components: `PascalCase.tsx` — e.g., `UserCard.tsx`, `BudgetDetailPage.tsx`
- Hooks: `camelCase` prefixed with `use` — e.g., `useAuthState.ts`, `useOnboardingErrorHandler.ts`
- Utilities/lib: `camelCase` — e.g., `formatCurrency.ts`, `dateUtils.ts`
- Redux slices: `camelCase` suffixed with `Slice` — e.g., `authSlice.ts`, `transactionSlice.ts`
- API modules: `camelCase` suffixed with `Api` — e.g., `categoryApi.ts`, `sessionApi.ts`
- Types files: `types.ts` per feature
- Constants files: `constants.ts` per feature
- Routes files: `routes.ts` per feature
- Page-level components named with `Page` suffix — e.g., `BudgetsPage.tsx`, `TransactionListPage.tsx`

### Naming Patterns (Backend)
- Classes: `PascalCase`
- Methods/variables: `camelCase`
- Packages: lowercase, domain-segmented — `com.kaizen.backend.<feature>.<layer>`
- DTOs: verb + noun + Request/Response — e.g., `BudgetCreateRequest`, `CategoryResponse`
- Exceptions: noun + `Exception` — e.g., `CategoryNotFoundException`, `DuplicateEmailException`

### Module / Import Style (Frontend)
- ESM (`"type": "module"` in `package.json`)
- `verbatimModuleSyntax` enabled — use `import type` for type-only imports
- Module resolution: `bundler` mode (no path aliases configured in `tsconfig.app.json`)
- Imports use relative paths; no `@/` aliases observed in tsconfig

### Commit Message Style
- Conventional Commits format observed in recent git log:
  - `feat(scope): description`
  - `fix(scope): description`
  - `docs: description`
- Evidence: commits `feat(home):`, `fix(home):`, `docs:` in recent history

### Branching Model
- Single protected branch: `main`
- PR-based workflow — CI triggers on PRs targeting `main` for both `frontend/**` and `backend/**` path filters
- Backend staging auto-deploys on push to `main`

### PR Template
Located at `.github/pull_request_template.md`. Fields:
- Description (1–2 sentences)
- Key technical implementations (bullet list)
- Checklist: Bug fix / New Feature / Documentation update / Other

## Constraints

### Mandatory (from AGENTS.md and CODING_STANDARDS.md)

**Global**
- `CODING_STANDARDS.md` is the single source of truth. It overrides all default style preferences in every task.
- Clarity over cleverness. Prefer readable, boring code.
- Small units: functions/classes do one thing. Backend methods target <= 40 lines.
- No silent failures — handle and log errors explicitly with context.
- Security by default: validate inputs, least privilege, never log secrets, passwords, tokens, or full PII payloads.

**TypeScript / Frontend**
- `strict: true` is required. No `any` in app code; prefer `unknown` + type guards.
- Avoid `enum`; prefer string unions.
- Use explicit return types on exported functions and public APIs.
- Feature-first organization (`features/*`) over type-first (`components/*`).
- `shared/` is only for truly reusable items.
- One styling system only: Tailwind CSS. No random inline styles.
- No arbitrary font sizes (`text-[15px]`), random gray/black text colors, or extra-bold weights in normal UI.
- Use approved typography roles only: `display`, `h1`, `h2`, `h3`, `h4`, `body-lg`, `body`, `body-sm`, `label`, `caption`, `code`.
- Use only three neutral text color levels: `text-foreground`, `text-muted-foreground`, `text-subtle-foreground`.
- Never use `text-black`, `text-white`, `text-gray-500`, `text-zinc-600`, or any raw palette token for text.
- Allowed font weights: `font-normal`, `font-medium`, `font-semibold` only.
- `tracking-tight` on headings only.
- WCAG AAA target for text content where practical; minimum `7:1` for normal text, `4.5:1` for large text.
- Non-text controls (borders, icons, focus rings) must meet at least `3:1` contrast.
- Theme token changes are not complete until both light and dark mode contrast is verified in the same change.
- Validation errors must not appear before field interaction (show on blur or submit, not on keystroke).
- Form fields must have visible labels (placeholders cannot replace labels).
- Environment variable additions must update the corresponding `.env.example` in the same change.

**Backend**
- Controllers must not talk directly to repositories.
- Services must not return JPA entities directly to controllers.
- All external inputs validated with `@Valid`, `@NotNull`, `@Size`, etc.
- Error responses use exactly two envelopes: `ErrorResponse` (general) or `ValidationErrorResponse` (field validation).
- `@Transactional` at service layer for atomic operations.
- Avoid `EAGER` fetch; prefer `LAZY` + explicit fetching.
- Use `IF NOT EXISTS` for migrations in development phases.
- Never edit a committed Flyway migration script — all changes are new, higher-numbered scripts.
- API documentation (OpenAPI/Swagger + Postman collection) must be updated in the same change as any endpoint addition or modification. Follow `backend/docs/API_DOCUMENTATION_STANDARDS.md`.
- Prefer constructor injection over field injection.
- Structured logging with `traceId`, user id, and operation name — never log secrets.

**Algorithm / Performance**
- State time complexity, space complexity, and expected input size for non-trivial logic.
- If `n` could exceed ~10k, O(n²) is unacceptable unless proven safe.
- Nested loops must be justified or optimized.
- Prefer pagination or streaming over loading large result sets into memory.

**PR Gate (both frontend and backend)**
- A PR cannot merge if lint, typecheck, or tests fail.
- CI checks: frontend — lint + build; backend — compile + SpotBugs + Docker build.
