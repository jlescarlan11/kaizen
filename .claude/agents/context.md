## Conventions

### Naming patterns
- React components: `PascalCase.tsx` — e.g. `TransactionListPage.tsx`, `DashboardCard.tsx`
- Hooks: `use` prefix camelCase — e.g. `useBalanceAnalytics.ts`, `useInsightsPeriod.ts`
- Utilities: camelCase — e.g. `formatCurrency.ts`, `dateUtils.ts`, `colorUtils.ts`
- Constants: `UPPER_SNAKE_CASE` — e.g. `SMART_BUDGET_PERIOD`, `ADD_TRANSACTION_ROUTE`
- Redux slices: noun + "Slice" — e.g. `transactionSlice`, `onboardingSlice`, `authSlice`, `uiSlice`
- RTK Query API files: noun + "Api" — e.g. `transactionApi`, `budgetApi`
- Page components: feature noun + "Page" — e.g. `BudgetsPage`, `InsightsPage`, `CategoryManagementPage`

## Constraints

### Hard rules from AGENTS.md (mandatory for all agents)

1. **CODING_STANDARDS.md is authoritative** over all code, architecture, testing, refactor, and review work. Conflicts resolve in favor of CODING_STANDARDS.md.

2. **Typography — use approved roles only** (defined in `frontend/src/shared/styles/typography.ts`):
   - Roles: `display`, `h1`, `h2`, `h3`, `h4`, `body-lg`, `body`, `body-sm`, `label`, `caption`, `code`
   - Tailwind utilities only — no arbitrary font sizes (e.g. no `text-[13px]`)
   - Only `font-normal`, `font-medium`, `font-semibold` permitted
   - Three-level neutral text token hierarchy only:
     - `text-foreground` — primary (headings, labels, body, important actions)
     - `text-muted-foreground` — secondary (descriptions, helper text)
     - `text-subtle-foreground` — tertiary (timestamps, metadata, microcopy)
   - Do not introduce a fourth neutral text level
   - `tracking-tight` only on headings; body text must be stable and consistent
   - Only headings should scale responsively

3. **Accessibility color policy:**
   - Target WCAG AAA for text content where practical
   - Filled colored surfaces: base, hover, and active states must all be contrast-checked
   - Borders, focus indicators, icons, controls: minimum `3:1` contrast ratio
   - Do not assume `text-foreground` is safe on a colored background without explicit verification
   - Any token or color change must be verified in both light and dark mode in the same PR

4. **Environment variable policy:** Adding or changing any env-var usage in backend, frontend, Docker, or workflows requires updating the corresponding `.env.example` in the same change.

5. **API documentation policy:** Any backend endpoint addition or change must include OpenAPI/Swagger doc updates, example/schema updates, and Postman collection template updates in the same PR. Follow `backend/docs/API_DOCUMENTATION_STANDARDS.md`.

### From CODING_STANDARDS.md
- `strict: true` TypeScript — no `any` in app code; use `unknown` + type guards instead
- No `enum` unless runtime values are needed; prefer string unions
- Explicit return types required for exported functions and public APIs
- One styling system only (Tailwind) — no inline styles except one-off dynamic values
- User-facing errors must be friendly, actionable, and not leak internal details
- Never swallow errors — log with operation name, entity ID, and user-safe context
- Security: validate all inputs, least privilege, never log secrets

## Known Issues and Pain Points

These are confirmed in source code as of Jun 3, 2026 and are directly relevant to UI/UX work:

1. **Goals feature is a complete stub.** `/goals` redirects to home; `/goals/:id` is a "Coming Soon" placeholder. The nav item appears live in the sidebar, creating a dead-end UX.

2. **`hover:bg-black/5` is a systemic dark-mode blocker** — confirmed across multiple shell components (Apr 2026 audit). Correct pattern is `hover:bg-surface-secondary`.

3. **`ActionCenterCard` uses raw Tailwind color classes** (`bg-amber-50`, `bg-green-50`, `bg-blue-50`, `border-amber-200`) — bypasses token system; dark-mode unsafe.

4. **`AppearancePage` uses `bg-white`** for theme option cards — not `bg-surface`; dark-mode unsafe.

5. **`HomeDashboardHeader` quick-action buttons are raw `<button>` elements** with hardcoded Tailwind classes — not using the `Button` component, inconsistent with component policy.

6. **`CategoryManagementPage` fetches via raw async `getCategories()` call** (local `api.ts`) instead of RTK Query — inconsistent data-fetching pattern vs. all other pages.

7. **No frontend CI.** No GitHub Actions workflow for frontend lint, typecheck, or build. TypeScript errors can reach `main` undetected.

8. **No automated test suite** on the frontend — verification relies entirely on local lint + typecheck.

9. **Two header implementations with different heights** — `SiteHeader` (`h-24`, unauthenticated) vs. authenticated top bar (`h-12`). Separate components, no shared abstraction.

10. **Breadcrumb is a direct flex child of the authenticated header** with no wrapper element — limits layout flexibility.

11. **Form input background uses `bg-white`** in some components — not semantic `bg-surface`; pending migration (flagged Jun 3, 2026).

12. **Type-filter chip close button missing `aria-label`** in `TransactionFilter` — other filter chips have it; accessibility gap.
